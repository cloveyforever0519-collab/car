## 1. 总体工作结论

截至 2026-05-28，本项目已经从早期多脚本、多链路、热修改较多的试验状态，收敛为以 CARLA 官方 pygame 主画面、HTTP 网关、systemd 用户服务、Traffic Manager 导航、传感器摘要输出和三路 RTSP 视频流为核心的交付形态。

工作重点不是单一功能开发，而是把 CARLA 仿真、AIGO 算法、Manual/CAN、前端接口、视频流、传感器和现场部署整合到一套能稳定运行、能解释、能验收、能恢复的系统中。尤其在最后两天，围绕路线急转、路线重复、后视镜视频延迟、全景摄像头、旧包热更改丢失、systemd 冲突、mediamtx 缺失、权限不足、高 GPU 功耗和传感器误测等问题完成了集中闭环。

当前最终成果包括：

- 形成 V12 final stable 最终包，明确禁止继续使用 V10/V11 旧包和 V13/V13.1 试验包。
- 后端主链路统一为 `official_carla_demo`。
- 三个 systemd 服务稳定管理 CARLA 引擎、后端网关和视频流。
- 普通 AI/AIGO/Manual 模式保留并可通过 `/command` 启动。
- `/view` 支持运行中切换第一/第三视角。
- `/route` 支持 Town02 AB/BC/CA 地图导航，使用 Traffic Manager fixed path，路线固定可复现。
- 传感器通过 `loadingsensor=0` 按需加载，并输出前端可直接展示的摘要字段。
- 左后视镜、右后视镜、车顶全景视角三路 RTSP 可同时拉流。
- 完成上行接口、下行接口、使用说明书、传感器字段说明和工作量总结文档。

## 2. 架构收敛工作

早期项目中存在多个历史链路：原始 GUI、native demo、RTSP 主画面、center-display、watchdog、临时脚本和多版本交付包。为降低现场维护复杂度，本次工作将交付主线收敛为：

```text
前端/控制端 -> HTTP 8765 official_demo_gateway.py
             -> official_udp_vehicle_client.py
             -> CARLA_0.9.15
             -> pygame 主画面 + UDP 遥测 + SensorSuite
             -> carla_mirror_stream.py + mediamtx 三路 RTSP
```

架构收敛带来的价值：

- 前端只需要围绕 8765 和 8554 两个核心入口对接。
- 主画面不再依赖二次编码 RTSP，减少延迟和故障点。
- 视频流职责从主画面中剥离，专门服务后视镜和全景视角。
- 传感器数据与车辆遥测解耦，避免点云/传感器数据冲击旧接口。
- systemd 管理统一，部署、重启、日志、排障路径清楚。

完成内容：

- `/view rear` 在主 pygame 画面中会归一到 follow 或兼容模式，真正左右后视镜不是 `/view`，而是 `side_camera_streams` 三路 RTSP。

## 6. 地图导航工作量

地图导航是最近两天工作量最大的部分之一。用户明确指出：转弯大不是速度问题，而是转角给得不自然；固定小转角也不是正确方案，因为真实车辆转弯会动态变化。并且路线不能 AB 经过 C、BC 经过 A、CA 经过 B，也不能大量重复同车道，否则前端画图和验收都受影响。

### 6.1 问题演进

| 阶段 | 方案 | 现象 | 结论 |
|---|---|---|---|
| 早期端点导航 | BasicAgent 直接从起点到终点 | AB/BC 转弯过猛，路线不理想 | 不适合作为最终导航主方案 |
| 降速尝试 | 降低速度 | 体验略改善但本质未解决 | 用户判断正确，不是速度核心问题 |
| 固定小转角尝试 | 限制 steer | 过弯不真实，不能适应路况 | 放弃作为最终方案 |
| Traffic Manager fixed path | CARLA 自带 TM 驾驶固定 path | 转弯自然，路线可固定 | 定为最终方案 |

### 6.2 最终实现

最终方案：

- 使用 `GlobalRoutePlanner` 基于真实道路生成 path。
- 构造 A/B/C 三点时，惩罚经过第三点、路线重叠和重复车道。
- 使用 `TrafficManager.set_path(vehicle, path)` 驱动车辆。
- 设置 seed=42，保证复现。
- 关闭自动变道。
- 输出 `path_summary.hash`，用于确认路线每次一致。
- `OFFICIAL_ROUTE_TM_SPEED_DIFFERENCE_PERCENT=0`，解决速度太慢。
- `OFFICIAL_ROUTE_ARRIVAL_DISTANCE_M=2.0`，解决弯没拐完提前停车。

### 6.3 验证结果

用户实际测试反馈：

- 路线选择可以接受。
- 过弯目前没问题。
- 速度调到正常限速后体验更好。
- BC、CA 可继续单独观察。

最终仍建议路线验收一条一条跑，不要写过快循环，避免“BC 还没跑完就切 CA”的人为误判。

## 7. 视频流与全景摄像头工作量

视频流需求从左右后视镜扩展到“车中心正上方的全景摄像头/上帝视角”，并要求三路理论上可以同时拉取，不能拉一路影响另一路。

完成内容：

- 增加 `carla_birdview` RTSP 流。
- 保留 `carla_rear_left`、`carla_rear_right`。
- 三路均为 1280x720、20fps、H264。
- mediamtx 内置到 V12 包的 `bin/mediamtx`，避免现场缺二进制。
- ffmpeg 优先使用 `h264_nvenc`，不可用时可回落。
- 使用最新帧低延迟队列，过期帧丢弃。
- `/health.side_camera_streams` 返回三路地址，前端按字段绑定。

甲方提到左右后视镜不同步、延迟 1-2 秒。处理思路不是只降画质，而是综合：

- 降到 1280x720、20fps，降低编码压力。
- 队列为 1，只取最新帧。
- 丢弃超过 0.35 秒旧帧。
- mediamtx 统一发布，避免多服务不一致。
- 前端播放器也需要低缓存参数。

最终用户拉流后检查结果：

- `publishing=true`
- `last_error=null`
- 三路 `ffprobe` 均 OK。
- 编码为 H264，分辨率 1280x720，帧率 20/1。

## 8. 高 GPU 功耗与 Vulkan 问题工作量

现场发现 CARLA 原生 Unreal 窗口即使空载也可能出现 GPU 95%-99%、功耗约 300W，并伴随 Vulkan 相关崩溃风险。曾讨论过 NVIDIA power limit，但最终不作为正式方案，因为它只是硬件限功耗，不解决 Unreal/Vulkan 呈现层问题。

尝试和取舍：

| 方案 | 结论 |
|---|---|
| NVIDIA power limit | 不作为正式方案 |
| t.MaxFPS / VSync / pygame 节奏 | 未能稳定解决核心问题 |
| motion blur 调整 | 用户反馈收益不明显，取消 |
| CARLA windowed + vulkanpresentmode=2 | 定为最终方案 |

最终 systemd drop-in 固化：

```text
CARLA_EXTRA_ARGS=-windowed -vulkanpresentmode=2
CARLA_FULLSCREEN=0
CARLA_RESOLUTION=1920x1080
CARLA_QUALITY_LEVEL=Low
```

实际服务状态可看到：

```text
CarlaUE4.sh -nosound -quality-level=Low -ResX=1920 -ResY=1080 -windowed -vulkanpresentmode=2
```

该工作避免了把问题转嫁给硬件功耗限制，也避免了因为临时 drop-in 丢失导致最终包恢复后问题复现。

## 12. 状态文件与并发稳定性工作量

联调中出现过 `official_gateway_status.tmp -> official_gateway_status.json` 的偶发错误。问题根因是多个线程/请求使用同一个 `.tmp` 临时文件名，存在并发替换竞争。

处理方式：

- `write_json` 改为 PID + 线程 ID + time_ns 组成唯一临时文件名。
- 写完后使用 `os.replace` 原子替换。
- 避免多个请求同时写同一个 tmp。

价值：

- 降低 `/route`、`/view`、`/health` 并发访问下的状态写入误报。
- 避免前端看到偶发 `ok:false`。
- 提高长时间运行稳定性。

## 16. 工作价值总结

本阶段工作价值主要体现在：

- 把零散功能收敛为可交付系统，而不是停留在脚本能跑。
- 解决了影响现场演示的核心稳定性问题：高 GPU、视频流、导航急转、旧包丢热更改。
- 把前端需要的新增能力做成明确接口：视角、传感器、导航、视频流。
- 把传感器从“配置说明”推进到“真实挂载+摘要输出”。
- 把地图导航从“不自然端点驾驶”推进到“固定 path + Traffic Manager 动态驾驶”。
- 把部署从“手动热操作”推进到“V12 最终包 + 一键安装 + systemd 固化”。
- 把验收从“凭眼看”推进到“服务状态、端口、health、ffprobe、日志、CSV 压测记录”。

一句话总结：本阶段不仅完成了功能，还把项目从“能演示”推进到了“能交付、能恢复、能排障、能让前端继续对接”的状态；最终收口后，交付包、交付机热更和文档口径已经统一。
