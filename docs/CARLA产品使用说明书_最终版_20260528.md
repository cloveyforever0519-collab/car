## 1. 产品定位与交付边界

本产品是一套基于 CARLA 0.9.15 的智能座舱/车辆仿真演示与联合仿真平台。系统的核心目标不是单纯打开一个 CARLA 地图，而是把前端大屏、CARLA 官方仿真、AIGO 算法控制、Manual/CAN 硬件控制、传感器数据、地图导航和多路视频输出组织成可部署、可验收、可维护的完整链路。

当前最终定版以 `official_carla_demo` 为主链路。主画面采用 CARLA 官方 pygame 方式显示，避免旧 RTSP 主画面方案带来的二次编码延迟和稳定性问题；左右后视镜和车顶全景摄像头作为独立 RTSP 流输出，供中控大屏或前端页面同时拉取播放。

最终交付范围包含：

- 普通驾驶模式：AI、AIGO、Manual 三类模式。
- 地图导航模式：Town02 的 AB、BC、CA 三段固定路线，采用 CARLA Traffic Manager fixed path 驾驶。
- 传感器功能：LiDAR、Radar、GNSS、IMU、超声/Obstacle 摘要输出，并保留主 RGB 摄像头、左右后视镜和车顶全景摄像头。
- 视频流功能：左后视镜、右后视镜、上帝/全景视角三路 RTSP，默认 1280x720、20fps、H264。
- 对外接口：HTTP 8765 上行控制和下行状态，UDP 5000/5002/5003 车辆遥测，UDP 5010 传感器摘要。
- 部署运行：systemd 用户服务自启动，V12 最终包一键安装，旧热配置冲突清理。
- 验收工具：服务状态检查、三路 RTSP 拉流检查、传感器 1 分钟检查、24 小时高压测试脚本。

不属于当前 V12 已交付功能但已评估过的内容：

- 车内发动机/环境音效：当前 CARLA 引擎以 `-nosound` 启动，V12 未交付音效服务。如后续甲方要求，可外置音频服务，根据油门开度、速度、转速估算或 CAN 信号驱动音频循环混音。
- 完整 LiDAR 点云大屏实时渲染：当前对外交付的是摘要、统计和少量样本，不把完整大点云塞进旧车辆遥测链路，以免影响主画面和控制稳定性。
- 前端页面最终 UI：后端接口已经具备，前端仍需要加入视角、传感器、导航、三路视频和传感器数据展示控件。

## 3. 系统总体架构

系统采用“HTTP 网关 + 官方 pygame 客户端 + CARLA 引擎 + 独立 RTSP 视频服务 + UDP 遥测”的架构。

| 层级 | 组件 | 主要职责 | 关键端口/文件 |
|---|---|---|---|
| 前端/中控/控制端 | Web 页面、curl、Postman、外部控制器 | 下发场景、模式、视角、传感器、路线、停止等命令；读取状态和视频流 | HTTP 8765、RTSP 8554 |
| HTTP 网关 | official_demo_gateway.py | 解析上行 JSON，启动/停止官方客户端，聚合健康状态，暴露 /health、/telemetry、/sensors、/route | /home/zhang/Carla_Project/official_carla_demo/official_demo_gateway.py |
| 官方 pygame 客户端 | official_udp_vehicle_client.py | 连接 CARLA，创建车辆、摄像头、传感器、导航路线，渲染主画面，输出 UDP 遥测 | official-pygame://camera |
| CARLA 引擎 | CARLA_0.9.15/CarlaUE4.sh | 地图、物理、Traffic Manager、传感器仿真、Unreal 渲染 | RPC 2000 |
| 视频服务 | carla_mirror_stream.py + mediamtx + ffmpeg | 生成左右后视镜和上帝视角三路 RTSP，支持同时拉流 | RTSP 8554 |
| 传感器输出 | SensorSuite | 输出 LiDAR、Radar、GNSS、IMU、超声/Obstacle 摘要 | UDP 5010、/sensors |
| Manual 硬件桥接 | can_tcp_bridge_vcu.py、can_wheel_bridge_vcu.py | TCP-CAN/DBC 解码，转为 CARLA Manual 控制 | TCP-CAN 192.168.110.112:4001、UDP 5001 |
| 服务管理 | systemd user services | 引擎、后端、视频服务自启动和崩溃重启 | ~/.config/systemd/user |

核心原则：

- 主画面不走 RTSP，使用官方 pygame 窗口，减少主画面延迟和编码链路故障。
- 左右后视镜和全景视角独立 RTSP，可与主画面同时存在。
- 地图导航不使用手写固定小转角，而借用 CARLA Traffic Manager 的动态驾驶能力。
- 传感器数据做摘要输出，避免完整点云冲击 UI、UDP 和仿真主循环。
- 部署以 V12 包为唯一基准，不再依赖零散热更改。

## 4. 运行环境要求

### 4.1 Ubuntu 与图形桌面

交付机要求为 Ubuntu 图形桌面环境，用户为 `zhang`。系统依赖 X11/GDM 会话，systemd 用户服务通过当前用户启动 CARLA。

必须满足：

- `/tmp/.X11-unix/X*` 图形 socket 存在。
- `/run/user/1000/gdm/Xauthority` 或 `~/.Xauthority` 可读。
- 用户服务可用：`systemctl --user status` 能正常执行。
- 建议开启用户自动登录，避免开机后没有图形会话导致 CARLA 无法启动。

检查命令：

```bash
echo "$DISPLAY"
ls -l /tmp/.X11-unix
ls -l /run/user/$(id -u)/gdm/Xauthority ~/.Xauthority 2>/dev/null || true
systemctl --user status --no-pager | head
```

### 4.2 CARLA 0.9.15 环境

CARLA 安装目录固定为：

```bash
/home/zhang/CARLA_0.9.15
```

引擎启动脚本为：

```bash
/home/zhang/CARLA_0.9.15/CarlaUE4.sh
```

V12 最终启动参数中必须能看到：

```text
-nosound -quality-level=Low -ResX=1920 -ResY=1080 -windowed -vulkanpresentmode=2
```

其中：

- `-nosound`：避免无人值守服务启动时音频设备不可用导致报错。
- `-quality-level=Low`：保持画质。
- `-ResX=1920 -ResY=1080`：主渲染尺寸。
- `-windowed`：避免 Unreal 全屏/呈现层导致空载 GPU 长期满载。
- `-vulkanpresentmode=2`：降低 Vulkan present 引发的高功耗和崩溃风险。

检查命令：

```bash
systemctl --user status carla-engine.service --no-pager -l | sed -n '1,45p'
```

### 4.3 conda 与 carla_vcu 环境

后端脚本通过 `delivery/lib.sh` 和 `official_carla_demo/runtime/lib.sh` 自动加载 conda。加载顺序为：

1. 如果 `conda` 命令可用，使用 `conda info --base`。
2. 否则检查 `~/miniconda3`。
3. 否则检查 `~/anaconda3`。
4. 激活 `CONDA_ENV`，默认值为 `carla_vcu`。

默认环境：

```bash
conda activate carla_vcu
```

需要具备的能力：

- 能导入 CARLA Python API。
- 能导入 pygame、numpy、Pillow。
- Manual/CAN 路径需要 cantools。
- ffmpeg/ffprobe 命令可用。

建议检查：

```bash
source "$HOME/miniconda3/etc/profile.d/conda.sh"
conda activate carla_vcu
python - <<'PY'
import sys
print(sys.executable)
for name in ["carla", "pygame", "numpy", "PIL"]:
    try:
        __import__(name)
        print(name, "OK")
    except Exception as exc:
        print(name, "FAIL", exc)
try:
    import cantools
    print("cantools OK")
except Exception as exc:
    print("cantools optional/manual path FAIL", exc)
PY
ffmpeg -version | head -n 1
ffprobe -version | head -n 1
```

CARLA PythonAPI 路径由后端 drop-in 注入：

```text
CARLA_EXAMPLES_DIR=%h/Workspace/carla_hil_project/PythonAPI/examples
CARLA_PYTHONAPI=%h/Workspace/carla_hil_project/PythonAPI/carla
```

如果 `GlobalRoutePlanner` 或 `BasicAgent` 导入失败，应检查上述路径是否存在，或检查 CARLA PythonAPI 是否与 0.9.15 匹配。

### 7.2 CARLA 引擎关键参数

| 变量 | 最终值 | 作用 |
|---|---|---|
| CARLA_RENDER_OFFSCREEN | 0 | 使用可见窗口，不采用离屏渲染 |
| CARLA_FULLSCREEN | 0 | 不使用 CARLA Unreal 全屏 |
| CARLA_RESOLUTION | 1920x1080 | CARLA 引擎分辨率 |
| CARLA_QUALITY_LEVEL | Low | 交付机低功耗稳定配置 |
| CARLA_EXTRA_ARGS | -windowed -vulkanpresentmode=2 | 解决空载高 GPU/Vulkan present 问题 |

现场曾出现 CARLA 原生 Unreal 窗口空载 GPU 长期 95%-99%、功耗约 300W、并伴随 Vulkan 崩溃风险。最终不采用 NVIDIA power limit 作为正式方案，而是用 `Low + windowed + vulkanpresentmode=2` 的 CARLA/Unreal 启动参数治理呈现层问题。

### 11.4 后视镜不同步问题的处理思路

甲方反馈左右后视镜可能不同步，左侧先动、右侧延迟 1-2 秒。最终处理不只靠降画质，而是组合处理：

- 分辨率调整为 1280x720。
- 帧率调整为 20fps。
- H264 码率 5000k。
- 队列长度为 1，只保留最新帧。
- 超过 0.35 秒的旧帧丢弃。
- 使用 TCP RTSP 拉流减少网络抖动下的花屏。
- mediamtx 本地发布三路，避免每路自建服务。

如前端仍感到延迟，应先确认播放器是否启用了缓存。建议播放器端也使用低延迟参数，而不是只改后端画质。

## 12. 地图导航模式说明

### 12.1 设计目标

地图导航需求包含几个硬要求：

- 每次同一路线要一致，不能每次随机跑。
- AB 不经过 C，BC 不经过 A，CA 不经过 B。
- 尽量减少同车道重复路线，方便前端画图。
- 转弯不能“固定死小角度”，真实车辆会先小角度、再大角度、再回正。
- 不能因为降低速度掩盖转角过大的问题。
- 弯没有拐完不能提前停车。

早期 BasicAgent 端点导航会出现急转和路线不理想问题。最终改为：

1. 使用 `GlobalRoutePlanner` 计算真实路网 waypoint path。
2. 选择 A/B/C 三点时惩罚经过第三点、路线重叠和重复车道。
3. 使用 `TrafficManager.set_path(vehicle, path)` 固定路线。
4. 设置 Traffic Manager seed=42。
5. 关闭自动变道。
6. 设置 `OFFICIAL_ROUTE_TM_SPEED_DIFFERENCE_PERCENT=0`。
7. 设置 `OFFICIAL_ROUTE_ARRIVAL_DISTANCE_M=2.0`。

这样既借用了 CARLA 自带 AI/Traffic Manager 的动态转向风格，又保证固定路线可复现。

### 12.2 路线状态字段

`/health.route` 关键字段：

| 字段 | 含义 | 验收标准 |
|---|---|---|
| enabled | 是否处于导航模式 | true |
| mode | AB/BC/CA | 与请求一致 |
| active_leg | 当前路线段 | 与请求一致 |
| target | 目标点 | AB 为 B，BC 为 C，CA 为 A |
| arrived | 是否到达 | 运行中 false，到达后 true |
| controller | 控制器 | 应为 traffic_manager_path |
| traffic_manager_message | TM 信息 | Traffic Manager set_path active for XX |
| target_speed_kmh | 目标/配置速度 | 状态展示 |
| tm_seed | 随机种子 | 42 |
| points | A/B/C 坐标 | 前端画图 |
| path_summary | 路线 hash 和 waypoint_count | 校验路线一致性 |
| route_lengths_m | 路网路径长度 | 调试/验收 |
| shared_route_cells | 路线重叠格统计 | 调试 |
| error | 错误信息 | 正常应为 null |

### 12.3 路线是否每次一样

同一版本代码、同一 Town02 地图、同一 Traffic Manager seed、同一 A/B/C 点选择逻辑下，路线应保持一致。前端或验收脚本可用 `route.path_summary.<segment>.hash` 做一致性校验。

如果 hash 变化，优先检查：

- 是否换了 CARLA 地图版本。
- 是否换了最终包版本。
- 是否清理/修改了路线参数。
- 是否用的是旧 V10/V11 包。
- 是否 Traffic Manager fixed path 降级为 fallback。

### 12.4 单段路线建议测试方式

路线验收不要写太快的 for 循环。推荐一条一条跑：

```bash
curl -sS -X POST http://127.0.0.1:8765/route \
  -H "Content-Type: application/json" \
  -d '{"scene":"Town02","segment":"BC","camera_view":"driver","loadingsensor":"1"}' \
  | python3 -m json.tool
```

然后观察 60-90 秒，再切下一条。过快切换会造成“BC 还没跑完就切 CA”的误判。

## 13. Manual/CAN/VCU 环境说明

Manual 模式用于和硬件驾驶舱/VCU/CAN 设备联调。当前保留两类桥接脚本：

- `can_tcp_bridge_vcu.py`
- `can_wheel_bridge_vcu.py`

### 13.1 TCP-CAN 默认环境

| 变量 | 默认值 | 说明 |
|---|---|---|
| CARLA_TCP_CAN_HOST | 192.168.110.112 | TCP-CAN 设备 IP |
| CARLA_TCP_CAN_PORT | 4001 | TCP-CAN 设备端口 |
| CARLA_TCP_CAN_RECONNECT_SEC | 3.0 | 断线重连间隔 |
| CARLA_CAN_DBC | can/NJ0515.dbc 或自动搜索 | DBC 文件 |
| MANUAL_BRIDGE_STATUS_FILE | logs/manual_bridge_status.json | Manual 桥状态 |
| CARLA_MANUAL_CTRL_PORT | 5001 | Manual 控制输入到 CARLA |
| CARLA_MANUAL_TELEM_PORT | 5000 | Manual 遥测接收 |

V12 安装脚本会为可能乱码的中文 DBC 文件创建 ASCII 别名：

- `can/NJ0515.dbc`
- `NJ0423.dbc`

这样可以避免 Windows/微信/zip 传输后中文文件名在 Linux 上乱码导致 DBC 找不到。

### 13.2 控制字段映射

`can_tcp_bridge_vcu.py` 从 `Cockpit_Control` 解码控制字段，并转换成 CARLA UDP 控制：

| CAN/DBC 字段 | 含义 | 转换 |
|---|---|---|
| Cockpit_ACC | 油门百分比 | throttle = ACC / 100 |
| Cockpit_Beak | 制动百分比 | brake = Brake / 100 |
| Cockpit_EPS_Angle | 方向盘角度 | steer = angle / 540，限幅 -1 到 1 |
| Cockpit_Gear | 档位 | 4=R，2/3=D，1=N，0=P，可通过环境变量调整 |

档位默认：

```text
CARLA_CAN_REVERSE_GEARS=4
CARLA_CAN_DRIVE_GEARS=2,3
CARLA_CAN_STATE_GEAR_NEUTRAL=1
CARLA_CAN_STATE_GEAR_DRIVE=2
CARLA_CAN_STATE_GEAR_REVERSE=4
```

如果现场倒档、D 档、P/N 识别不一致，应优先改环境变量或 DBC 映射，不要直接改主仿真代码。

### 15.3 24 小时高压测试

该脚本会记录主日志 `stress_24h.log`、CSV 采样 `stress_24h_samples.csv` 和失败日志 `stress_24h_failures.log`。它先 STOP，再启动 Town04 AI 第一视角带传感器场景，避免 warming up 导致 START 被忽略。

```bash
bash <<'SH'
set -euo pipefail

BASE="http://127.0.0.1:8765"
LOG_DIR="$HOME/Carla_Project/logs/stress_24h_$(date +%Y%m%d_%H%M%S)"
mkdir -p "$LOG_DIR"

MAIN_LOG="$LOG_DIR/stress_24h.log"
CSV_LOG="$LOG_DIR/stress_24h_samples.csv"
FAIL_LOG="$LOG_DIR/stress_24h_failures.log"

DURATION_SEC=$((24 * 60 * 60))
INTERVAL_SEC=60
RTSP_CHECK_EVERY=10
VIEW_SWITCH_EVERY=15

echo "timestamp,elapsed_sec,running,vehicle_alive,world,view,speed_kmh,sensor_enabled,lidar_points,radar_targets,nearest_obstacle_m,imu_accel_norm_m_s2,heading_deg,rear_left_ok,rear_right_ok,birdview_ok" > "$CSV_LOG"

log() { echo "[$(date '+%F %T')] $*" | tee -a "$MAIN_LOG"; }
fail() { echo "[$(date '+%F %T')] $*" | tee -a "$FAIL_LOG" "$MAIN_LOG"; }

rtsp_check() {
  local url="$1"
  timeout 8 ffprobe -rtsp_transport tcp -v error \
    -select_streams v:0 \
    -show_entries stream=codec_name,width,height,r_frame_rate \
    -of csv=p=0 "$url" >/dev/null 2>&1
}

log "=== 24小时高压测试开始 ==="
log "日志目录：$LOG_DIR"

for svc in carla-engine.service carla-backend.service carla-video.service; do
  state="$(systemctl --user is-active "$svc" 2>/dev/null || true)"
  log "$svc = $state"
  [ "$state" = "active" ] || fail "$svc not active"
done

log "停掉当前 demo，避免 warming up 导致启动被忽略"
curl -sS -X POST "$BASE/command" \
  -H "Content-Type: application/json" \
  -d '{"sendstate":"STOP"}' >> "$MAIN_LOG" 2>&1 || true
sleep 6

log "启动 Town04 / AI / 第一视角 / 传感器开启"
curl -sS -X POST "$BASE/command" \
  -H "Content-Type: application/json" \
  -d '{
    "scene":"Town04",
    "weather":"Sunny",
    "time_of_day":"Noon",
    "mode":"AI",
    "vehicle":"Tesla Model 3",
    "camera_view":"driver",
    "loadingtransportation":"1",
    "loadingsensor":"0"
  }' | tee -a "$MAIN_LOG"

sleep 30

START_TS="$(date +%s)"
COUNT=0
FAIL_COUNT=0

while true; do

  COUNT=$((COUNT + 1))
  TS="$(date '+%F %T')"
  TMP="$(mktemp)"

  if ! curl -sS "$BASE/health" > "$TMP"; then
    fail "health curl failed at ${ELAPSED}s"
    FAIL_COUNT=$((FAIL_COUNT + 1))
    rm -f "$TMP"
    sleep "$INTERVAL_SEC"
    continue
  fi

import sys,json
p,ts,elapsed=sys.argv[1:4]
h=json.load(open(p,encoding="utf-8"))
d=h.get("diagnostics") or {}
s=(h.get("sensor_data") or {}).get("frontend_summary") or h.get("sensor_summary") or {}
print(json.dumps({
  "ts": ts,
  "elapsed": int(elapsed),
  "running": h.get("running"),
  "vehicle_alive": h.get("vehicle_alive"),
  "world": h.get("world"),
  "view": h.get("camera_view"),
  "speed_kmh": d.get("speed_kmh"),
  "sensor_enabled": s.get("enabled"),
  "lidar_points": s.get("lidar_points"),
  "radar_targets": s.get("radar_targets"),

  "imu_accel_norm_m_s2": s.get("imu_accel_norm_m_s2"),
  "heading_deg": s.get("heading_deg")
}, ensure_ascii=False))
PY
)"

import sys,json
p,ts,elapsed=sys.argv[1:4]
h=json.load(open(p,encoding="utf-8"))
d=h.get("diagnostics") or {}
s=(h.get("sensor_data") or {}).get("frontend_summary") or h.get("sensor_summary") or {}
vals=[ts, elapsed, h.get("running"), h.get("vehicle_alive"), h.get("world"), h.get("camera_view"),
      d.get("speed_kmh"), s.get("enabled"), s.get("lidar_points"), s.get("radar_targets"),
      s.get("nearest_obstacle_m"), s.get("imu_accel_norm_m_s2"), s.get("heading_deg")]
print(",".join("" if v is None else str(v) for v in vals))
PY
)"
  rm -f "$TMP"

  REAR_LEFT_OK=""
  REAR_RIGHT_OK=""
  BIRD_OK=""

  if [ $((COUNT % RTSP_CHECK_EVERY)) -eq 0 ]; then
    rtsp_check "rtsp://127.0.0.1:8554/carla_rear_left" && REAR_LEFT_OK="1" || REAR_LEFT_OK="0"
    rtsp_check "rtsp://127.0.0.1:8554/carla_rear_right" && REAR_RIGHT_OK="1" || REAR_RIGHT_OK="0"
    rtsp_check "rtsp://127.0.0.1:8554/carla_birdview" && BIRD_OK="1" || BIRD_OK="0"
    [ "$REAR_LEFT_OK" = "1" ] || { fail "rear_left RTSP failed at ${ELAPSED}s"; FAIL_COUNT=$((FAIL_COUNT + 1)); }
    [ "$REAR_RIGHT_OK" = "1" ] || { fail "rear_right RTSP failed at ${ELAPSED}s"; FAIL_COUNT=$((FAIL_COUNT + 1)); }
    [ "$BIRD_OK" = "1" ] || { fail "birdview RTSP failed at ${ELAPSED}s"; FAIL_COUNT=$((FAIL_COUNT + 1)); }
  fi

  echo "$CSV_PART,$REAR_LEFT_OK,$REAR_RIGHT_OK,$BIRD_OK" >> "$CSV_LOG"

  if [ $((COUNT % VIEW_SWITCH_EVERY)) -eq 0 ]; then
    if [ $(((COUNT / VIEW_SWITCH_EVERY) % 2)) -eq 0 ]; then VIEW="driver"; else VIEW="follow"; fi
    log "切换视角：$VIEW"
    curl -sS -X POST "$BASE/view" \
      -H "Content-Type: application/json" \
      -d "{\"camera_view\":\"$VIEW\"}" >> "$MAIN_LOG" 2>&1 || fail "view switch failed: $VIEW"
  fi

  if echo "$LINE" | grep -q '"sensor_enabled": false'; then
    fail "sensor disabled at ${ELAPSED}s"
    FAIL_COUNT=$((FAIL_COUNT + 1))
  fi

  sleep "$INTERVAL_SEC"
done

log "=== 24小时高压测试结束 ==="
log "采样次数：$COUNT"
log "失败计数：$FAIL_COUNT"
log "主日志：$MAIN_LOG"
log "CSV记录：$CSV_LOG"
log "失败日志：$FAIL_LOG"

echo
echo "完成。日志在：$LOG_DIR"
SH
```

## 16. 常见问题与排障

| 现象 | 判断 | 处理 |
|---|---|---|
| 重新部署后 engine 又变成 Epic | 用了旧包或旧 drop-in | 使用 V12 final stable，确认 install_autostart.sh 写入 CARLA_QUALITY_LEVEL=Low |
| Cybertruck 第一视角对着车头 | 未合入最终 Cybertruck driver camera 热更 | 使用 V12 final stable，确认 `driver` profile 为驾驶员座舱位置 |
| V13/V13.1 一发新命令就崩 | 中途试验包不稳定 | 不再部署 V13/V13.1，回到 V12 final stable |
| curl 8765 连接被拒绝 | carla-backend.service 没起来 | 看 `journalctl --user -u carla-backend.service -n 120 --no-pager` |
| backend 报 runtime/start_backend.sh 权限不足 | 旧 V11 包或权限未设置 | 使用 V12，执行 `bash ~/Carla_Project/install_latest_final.sh` |
| video 服务报 mediamtx not found | 旧包缺 mediamtx 或路径不对 | 使用 V12，确认 `/home/zhang/Carla_Project/bin/mediamtx` 可执行 |
| V10 zip 解压警告 backslashes as path separators | Windows zip 路径分隔问题 | 不用 zip，使用 Linux tar.gz V12 |
| mirror status 显示 waiting for ego vehicle | 还没启动车辆 | 先发 `/command` 或 `/route` |
| 三路视频不同步 | 播放器缓存或队列延迟 | 后端已低延迟，前端播放器也要低缓存 |
| CARLA 空载 GPU 很高 | Unreal/Vulkan present 问题 | 确认 engine 命令含 `-windowed -vulkanpresentmode=2` |
| 路线转弯太猛 | 旧 BasicAgent 或手写控制方案 | 使用 V12 TM fixed path |
| 弯没拐完就停车 | 到达判定太大 | V12 为 `OFFICIAL_ROUTE_ARRIVAL_DISTANCE_M=2.0` |
| 路线太慢 | TM speed difference 不是 0 | V12 为 `OFFICIAL_ROUTE_TM_SPEED_DIFFERENCE_PERCENT=0` |
| route 还没跑完就切下一条 | 测试脚本切太快 | 一条一条测，每条观察 60-90 秒 |
| 传感器一直 false | loadingsensor=1 或 START ignored | 用 loadingsensor=0；先 STOP 等 6 秒再 START |
| /view rear 变 follow | rear 是主画面兼容别名 | 真正后视镜看 side_camera_streams |
| official_gateway_status.tmp 报错 | 旧网关 tmp race | V12 已用唯一 tmp + os.replace 修复 |
| Manual 档位不对 | DBC/档位枚举不一致 | 调整 CARLA_CAN_REVERSE_GEARS / CARLA_CAN_DRIVE_GEARS |

## 17. 声音/音效需求评估

当前 V12 交付包未启用音效，CARLA 引擎启动参数包含 `-nosound`。这是为了避免 systemd 无人值守启动时音频设备不可用造成服务失败。

如果后续甲方要求“根据油门开度配发动机声”或“手动驾驶时从 CAN 获取声音信号”，建议采用外置音频服务，而不是强行依赖 CARLA 内部声音：

| 方案 | 输入 | 输出 | 难度 | 说明 |
|---|---|---|---|---|
| 基于车辆遥测合成 | speed、throttle、brake、gear | 本机音频播放 | 中 | 最快可做，用 pygame mixer 或其他音频库 |
| 基于 CAN 信号合成 | Cockpit_ACC、Gear、可能的 RPM | 本机音频播放 | 中 | Manual 模式更真实，需要确认 CAN 是否有转速/声音相关信号 |
| 专业音频引擎 | 遥测/CAN -> FMOD/Wwise | 多层音效 | 高 | 更真实但工作量大 |
| 前端播放音效 | 后端推 throttle/speed，前端 WebAudio 播放 | 浏览器音频 | 中 | 中控屏易集成，但延迟和自动播放权限要处理 |

推荐演进路线：

1. 第一阶段：根据 `diagnostics.speed_kmh` 和油门开度做基础循环音效。
2. 第二阶段：Manual 模式加入 CAN 档位和油门信号。
3. 第三阶段：加入启动、刹车、碰撞、环境音、转向灯等事件音。

当前文档只记录评估，不把音效写成 V12 已交付能力。
